# План внедрения «сервер — источник истины» (Convex) и усиления FSD

## Этап 0. Подготовка (границы и вынос логики)
- Добавить в тип `DialogChoice` поле `condition?: string` (+ опц. `tags?: string[]`).
- Создать `src/features/quest-progress`:
  - `model/actionCoordinator.ts` — маппинг `actionKey -> команды стора/репозитория`.
  - `model/visibility.ts` — функции фильтрации видимых точек по шагам квестов.
- Вынести выбор `dialogKey` в `features` (`decideDialogKey`), фильтрацию `condition` — в `DialogModal`.
- Включить линтер архитектурных границ (eslint-plugin-boundaries/steiger).

Статус: выполнено. Добавлены `condition/tags`, вынесены `visibility/decideDialogKey`, координатор действий, подключён eslint-boundaries.

## Этап 1. Convex: схема и серверные функции
- Расширить `convex/schema.ts`:
  - `map_points`: { id, title, description, coordinates, type, dialogKey, questId, active }.
  - `quest_progress`: { userId/deviceId, questId, currentStep, startedAt, updatedAt, completedAt }.
- `convex/quests.ts`:
  - queries: `getProgress({ userId })`.
  - mutations: `startQuest`, `advanceQuest`, `completeQuest` (атомарность).
- `convex/mapPoints.ts`:
  - query: `listAll()` и (опц.) `listVisible({ userId })`.

Статус: выполнено. Добавлены таблицы `map_points`, `quest_progress`; реализованы `getProgress/start/advance/complete`, `listAll`. `startQuest` не перезаписывает завершённый квест.

## Этап 2. Клиентские адаптеры (репозитории)
- `shared/api/mapPoints` и `shared/api/quests` с реализациями `local` и `convex`.
- Подписки на Convex queries и вызов mutations (идемпотентность со стороны сервера).
- Переключатель реализации (env-флаг/конфиг).

Статус: выполнено частично. Добавлены Convex-адаптеры `shared/api/{mapPoints,quests}`; сейчас используется Convex, локальный API остаётся как dev-фоллбэк.

## Этап 3. Интеграция в UI

Статус: выполнено.
