# Localization Guide (i18n)

This project uses a hybrid localization system designed to be type-safe, performant, and easily maintainable.

## Architecture

1.  **Canonical Source**: The primary narrative content is stored in `.md` files and extracted into `pilot.snapshot.json`. These files contain English text and stable IDs.
2.  **Translation Dictionary**: Localized strings are stored in `src/features/i18n/locales/{lang}.json`.
3.  **Lazy Loading**: Translations are loaded on-demand using dynamic imports in `I18nProvider.tsx`.
4.  **Resolver**: `vnContentTranslations.ts` provides helper functions to map stable IDs from the snapshot to translated strings in the dictionary.

## How to Add Translations

### 1. Identify the Key

Keys follow a hierarchical structure:

- **VN Nodes**: `vn.{scenarioId}.{nodeId}.title` or `vn.{scenarioId}.{nodeId}.body`
- **VN Choices**: `vn.{scenarioId}.{nodeId}.choice.{choiceId}`
- **Origins**: `origin.{originId}.label`, `origin.{originId}.summary`, etc.
- **Speakers**: `speakers.{speakerId}` (e.g., `speakers.inner_leader`)
- **Stats**: `stats.{statKey}` (e.g., `stats.attr_logic`)

### 2. Update `ru.json`

Add the key and its translation to the corresponding section in `src/features/i18n/locales/ru.json`.

```json
{
  "speakers": {
    "inner_leader": "Лидер"
  },
  "vn": {
    "vn.case01_hbf_arrival.scene_start.title": "Прибытие"
  }
}
```

### 3. Use the Resolver in Components

Avoid accessing the dictionary directly. Use the provided helpers:

```typescript
const { language, dictionary } = useI18n();
const translatedBody = resolveVnNodeText(
  language,
  scenarioId,
  nodeId,
  "body",
  originalBody,
  dictionary,
);
```

## Maintenance

- **Fallback**: If a translation is missing, the system automatically falls back to the English text from the snapshot.
- **Shadow Tests**: If you see test failures in `data/viking/`, these are likely outdated artifacts. The primary tests are in `src/`.
- **Dynamic Content**: For content that includes variables (e.g., skill checks), ensure the key in `ru.json` matches the stable ID generated by the content pipeline.
