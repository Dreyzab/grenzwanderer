# Features (Фичи)

Эта директория содержит реализации специфических функциональностей приложения или пользовательских историй. Каждая фича обычно оркеструет взаимодействия между различными `entities` (сущностями), обрабатывает пользовательский ввод, управляет локальным состоянием UI и взаимодействует с бэкенд-сервисами (например, через Convex). Фичи часто реализуются в виде кастомных хуков React или компонентов.

## Структура фичи

Типичная поддиректория фичи может включать:

-   **`api/`**: Содержит кастомные хуки React, которые инкапсулируют логику фичи, управление состоянием (часто локальное для фичи или взаимодействующее с глобальными сторами Effector) и вызовы API.
    -   `use[FeatureName].ts`: Основной хук для фичи.
-   **`ui/`**: Компоненты React, специфичные для этой фичи.
-   **`model/`**: Модели Effector (сторы, события, эффекты), если фича имеет сложную логику состояния, которая выигрывает от использования Effector и должна быть разделяемой или реагировать на изменения в разных частях приложения.
-   **`lib/`**: Вспомогательные функции, специфичные для фичи.
-   **`types.ts`**: Определения TypeScript, специфичные для фичи.
-   **`index.ts`**: Публичный API фичи, реэкспортирующий ключевые хуки, компоненты или типы.

## Реализованные фичи

### `game` (Игра)
Оркестрирует главный игровой экран, управляя навигацией между различными игровыми видами (карта, диалог, сканер, новелла) и инициализируя основные игровые данные.
-   **`api/useGamePage.ts`**: Хук для общей игровой страницы, обрабатывающий навигацию по вкладкам, инициализацию данных игрока (через Convex) и результаты сканирования QR-кодов, которые могут запускать сцены визуальной новеллы. Включает откат к мок-данным для QR-сканирований.
-   **`api/useGameScreen.ts`**: Хук более высокого уровня, который интегрирует различные подфичи, такие как состояние игрока, прогресс квестов, сообщения и маркеры на карте, в единый игровой экран. Он управляет текущим `GameView` (карта, сообщения, новелла) и координирует действия.

### `markers` (Маркеры)
Предоставляет функциональность для взаимодействия с маркерами на карте, в основном путем связывания действий UI с моделью Effector сущности `markers`.
-   **`api/useMarkers.ts`**: Хук для доступа к данным маркеров из стора Effector и диспетчеризации действий, таких как переключение активного/завершенного состояния или добавление взаимодействий. Ранее также содержал логику обновления видимости маркеров на основе состояния квеста (хотя это консолидируется в модели Effector фичи квестов).

### `messages` (Сообщения)
Управляет сообщениями пользователя, включая новые сообщения, архив и статус непрочитанных, с сохранением в `localStorage`.
-   **`api/useMessages.ts`**: Хук на основе `useReducer` для управления списками новых и архивных сообщений, их статусом прочтения, а также сохранением/загрузкой из `localStorage` с ключом по `playerId`.
-   **`api/useUnreadMessages.ts`**: Хук для определения, есть ли у текущего игрока непрочитанные сообщения, путем проверки `localStorage`, с периодическим опросом.

### `player` (Игрок)
Предоставляет хуки для доступа и взаимодействия с данными, специфичными для игрока, и характеристиками персонажа.
-   **`api/useCharacterStats.ts`**: Хук для отображения и изменения RPG-атрибутов игрока (основных/вторичных) и навыков путем взаимодействия со сторами Effector, определенными в сущности `player`.
-   **`api/useInventory.ts`**: Хук для управления инвентарем игрока. Получает предметы инвентаря из Convex и объединяет их с (в настоящее время моковыми) деталями предметов. Предоставляет мутации для экипировки, использования и выбрасывания предметов.

### `quest` (Квест)
Управляет квестовой системой игры, включая переходы состояний, действия и интеграцию с QR-кодами.
-   **`model.ts`**: Модель Effector, определяющая конечный автомат состояний квеста, переходы и связанную логику, такую как обновление состояния сервера и управление видимостью маркеров на основе прогресса квеста.
-   **`model/mockData.ts`**: Предоставляет мок-данные для результатов сканирования QR-кодов и инициации квестов, используемые в качестве запасных вариантов.
-   **`api/useQuestActions.ts`**: Хук, предоставляющий функции для обработки действий, связанных с квестами, таких как успешное сканирование QR-кодов или начало нового квеста. Он взаимодействует с мутациями Convex и использует мок-данные в качестве запасного варианта. Он также координирует изменения сцен и обновление маркеров.
-   **`types.ts`**: Определяет типы TypeScript для действий и переходов квеста.

### `scanner` (Сканер)
Реализует функциональность сканирования QR-кодов с использованием камеры устройства.
-   **`api/useQRScanner.ts`**: Хук, который использует `jsQR` для декодирования QR-кодов из живого видеопотока камеры или захваченной фотографии. Управляет доступом к камере, состоянием сканирования и обрабатывает ручной ввод кода.

### `scene` (Сцена)
Управляет логикой и состоянием для сцен визуальной новеллы.
-   **`useSceneLoader.ts`**: Хук, отвечающий за загрузку данных сцены. Сначала пытается получить их из Convex, затем откатывается к предопределенным тестовым сценам или общей мок-сцене при необходимости.
-   **`useSceneStateManager.ts`**: Хук для управления динамическим состоянием *внутри* активной сцены визуальной новеллы, таким как временные изменения статов, сделанные выборы или предметы, полученные во время сцены. Может синхронизировать некоторые изменения (например, статы игрока) обратно с глобальными сторами Effector.
-   **`useSceneChoice.ts`**: Хук для обработки выбора игрока, сделанного в сцене визуальной новеллы. Он проверяет условия, выполняет связанные действия (например, обновление состояния квеста через Effector, выдача предметов) и определяет следующую сцену или выходит из режима визуальной новеллы.

## Ключевые принципы

-   **Разделение ответственностей (Separation of Concerns)**: Фичи строятся на основе `entities`, но инкапсулируют специфическую пользовательскую логику.
-   **Переиспользуемость**: Хуки спроектированы так, чтобы их можно было повторно использовать в различных частях UI, где требуется данная фича.
-   **Управление состоянием**: Используется смешанный подход: локальное состояние React (`useState`, `useReducer`) для специфичных для UI задач и Effector для более глобального или сложного состояния, пересекающего несколько фич.
-   **Взаимодействие с API**: Фичи часто обрабатывают коммуникацию с бэкендом (Convex) через мутации и запросы.
-   **Запасные варианты/Мокирование**: Несколько фич включают надежные механизмы отката к мок-данным, что помогает в разработке и повышает отказоустойчивость.

## Структура API-хуков

### Game (Игровые)

- **useGamePage** - Управление состоянием и логикой главной игровой страницы.
  - Отвечает за навигацию по вкладкам, инициализацию игрока, обработку QR-сканирования.
  - Взаимодействует с Convex API для получения данных игрока и активации квестов.
  - Содержит механизм отката к мок-данным при недоступности API.

- **useGameScreen** - Центральный оркестратор состояний и действий игрового экрана.
  - Объединяет логику игрока, квестов, сообщений и маркеров.
  - Управляет переключениями между представлениями (карта, роман, инвентарь).
  - Обрабатывает взаимодействие с интерактивными элементами игрового мира.

### Player (Игрок)

- **useCharacterStats** - Управление атрибутами и навыками персонажа.
  - Взаимодействует с моделями сущности player (Effector).
  - Предоставляет методы для распределения атрибутов и изучения навыков.
  - Проверяет доступность навыков на основе требований.

- **useInventory** - Управление инвентарем игрока.
  - Интегрируется с Convex для получения и изменения данных инвентаря.
  - Предоставляет функции для экипировки, использования и выбрасывания предметов.
  - Содержит механизм отката к мок-данным для тестирования.

### Quest (Квесты)

- **useQuestActions** - Обработка действий, связанных с квестами.
  - Взаимодействует с Convex API для обработки QR-сканирований и запуска квестов.
  - Интегрируется с моделью квестов Effector для обновления состояния.
  - Содержит механизм отката к мок-данным при недоступности API.

### Markers (Маркеры)

- **useMarkers** - Взаимодействие с маркерами на карте.
  - Подписывается на сторы Effector для маркеров и их взаимодействий.
  - Предоставляет методы для показа, скрытия и переключения состояния маркеров.
  - Обновляет видимость маркеров в зависимости от состояния квеста.

### Messages (Сообщения)

- **useMessagesReducer** - Локальное управление сообщениями игрока.
  - Использует useReducer для управления состоянием сообщений.
  - Сохраняет состояние в localStorage для персистентности.
  - Предоставляет методы для добавления, пометки прочитанными и архивации сообщений.

- **useUnreadMessages** - Проверка наличия непрочитанных сообщений.
  - Периодически проверяет localStorage для обнаружения новых сообщений.
  - Инициализирует игрока для получения playerId.
  - Предоставляет флаг наличия непрочитанных сообщений для UI-компонентов.

### Scanner (Сканер)

- **useQRScanner** - Работа со сканером QR-кодов.
  - Взаимодействует с камерой устройства через jsQR.
  - Предоставляет методы для захвата и расшифровки QR-кодов.
  - Поддерживает ручной ввод кода для тестирования.

### Scene (Сцены)

- **useSceneStateManager** - Управление состоянием внутри сцены визуальной новеллы.
  - Отслеживает локальный прогресс квеста, изменения статов и инвентаря.
  - Предоставляет интерпретатор для выполнения скриптов.
  - Проверяет условия для диалоговых опций.

- **useSceneLoader** - Загрузка данных сцены для визуальной новеллы.
  - Получает данные сцены из Convex API.
  - Содержит механизм отката к локальным мок-данным.
  - Адаптирует структуру данных с сервера к внутреннему формату.

- **useSceneChoice** - Обработка выбора игрока в диалогах.
  - Проверяет условия выбора с помощью sceneStateManager.
  - Выполняет действия, связанные с выбором (обновление статов, квеста, выдача предметов).
  - Обрабатывает навигацию между сценами.

## Принципы создания API-хуков

1. **Композиция**: Каждый хук имеет специфическую ответственность, более сложные хуки (например, useGameScreen) объединяют функциональность других хуков.

2. **Отказоустойчивость**: Хуки, взаимодействующие с внешними API, содержат механизмы отката к мок-данным при недоступности сервера.

3. **Разделение ответственности**: 
   - Модели Effector отвечают за управление состоянием
   - API-хуки предоставляют интерфейс для взаимодействия с этим состоянием
   - UI-компоненты используют хуки для получения и изменения данных

4. **Персистентность**: Критические данные (сообщения, инвентарь) сохраняются в localStorage для обеспечения работы даже при отсутствии соединения.
