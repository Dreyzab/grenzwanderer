# Shared Module (`src/shared`)

Этот каталог содержит код, который является общим и переиспользуемым для различных частей приложения "Grenzwanderer". Цель — избежать дублирования кода и обеспечить единый источник правды для констант, типов, UI-компонентов и утилит.

## Структура

-   **`/constants`**: Глобальные константы приложения.
    -   `marker.ts`: Константы, связанные с QR-кодами для маркеров на карте.
    -   `quest.ts`: Перечисления (`enum`) для состояний квестов и возможных действий игрока.
-   **`/hooks`**: Переиспользуемые React-хуки.
    -   `useLocalStorage.ts`: Хук для работы с localStorage с поддержкой типизации и debounce.
-   **`/services`**: Сервисы для взаимодействия с внешними API или инкапсуляции сложной бизнес-логики.
    -   `api.service.ts`: Централизованный сервис для всех запросов к бэкенду Convex, сгруппированных по доменам (квесты, игроки, инвентарь).
-   **`/types`**: Определения TypeScript-типов и интерфейсов.
    -   `gameScreen.ts`: Типы для различных игровых экранов и их пропсов.
    -   `marker.types.ts`: Типы для маркеров на карте, включая их виды, классы NPC и фракции.
    -   `quest.types.ts`: Типы для квестов, действий и состояний квестовой системы.
    -   `scanner.ts`: Типы пропсов для компонента QR-сканера.
    -   `shelter.ts`: Типы, связанные с убежищем игрока и системой крафта.
    -   `visualNovel.ts`: Детальные типы для системы визуальной новеллы (персонажи, сцены, диалоги, выборы, статистика игрока).
-   **`/ui`**: Общие, переиспользуемые UI-компоненты.
    -   `/Button`: Компонент кнопки с различными вариантами, размерами и состояниями.
    -   `/Card`: Компонент карточки для отображения структурированного контента.
    -   `/ControlPanel`: Компонент панели управления для интерфейса приложения.
    -   `/DialogHistoryPanel`: Панель для отображения истории диалогов.
    -   `/Input`: Компонент поля ввода с поддержкой лейблов, иконок и состояний ошибки.
    -   `/Modal`: Компонент модального окна с анимациями и настройками отображения.
    -   `/PageLayout`: Компонент-обертка для стандартной структуры страниц приложения.
    -   `/SceneEffects`: Компонент для отображения визуальных эффектов на сцене (например, дождь, туман).
    -   `/Tooltip`: Компонент для отображения всплывающих подсказок.
    -   `ControlPanel.tsx`: Основной компонент панели управления для интерфейса визуальной новеллы.
    -   `DialogHistoryPanel.tsx`: Компонент для отображения истории диалогов.
    -   `index.ts`: Файл-агрегатор для экспорта всех UI-компонентов из одной точки.
    -   `theme.css`: Ключевой файл дизайн-системы, определяющий CSS-переменные для всего приложения. Включает:
        -   Основные цвета приложения (фон, поверхности, текст, акцентные цвета)
        -   Параметры типографики (шрифты, размеры, интерлиньяж)
        -   Отступы и размеры (spacing)
        -   Радиусы скругления
        -   Тени и эффекты (включая неоновые эффекты свечения)
        -   Анимационные параметры (скорость переходов, функции смягчения)
        -   Z-индексы для слоев интерфейса
        -   Поддержку светлой темы через атрибут `data-theme="light"`
    -   `tailwind.css`: Дополнительные стили и анимации для расширения функциональности Tailwind CSS:
        -   Кастомные @keyframes анимации (fadeIn, pulse, blink)
        -   Утилитарные классы для анимаций (animate-fade-in, animate-pulse-opacity)
        -   Классы для позиционирования персонажей (character-left, character-center, character-right)
        -   Общие стили для компонентов, которые не покрываются стандартным Tailwind
    -   `README.md`: Дополнительная документация по UI-компонентам.
-   **`/utils`**: Вспомогательные функции и утилиты.
    -   `convex.tsx`: Инициализация и экспорт клиента Convex, включая React-компоненты для интеграции Convex с приложением.
    -   `mapUtils.ts`: Утилиты для работы с картой, генерации маркеров и расчета расстояний.

## Принципы

-   **Переиспользование:** Код в `shared` должен быть достаточно общим для использования в нескольких местах.
-   **Низкая связанность:** Компоненты и утилиты из `shared` не должны зависеть от конкретных фичей приложения.
-   **Единый стиль:** Все UI-компоненты должны следовать дизайн-системе, определенной в `theme.css`, и использовать Tailwind CSS, настроенный на работу с этими CSS-переменными.

## Стилизация и темизация

Проект использует комбинацию подходов для стилизации:

1. **CSS-переменные:** Определены в `theme.css` и представляют собой единый источник правды для всех стилевых параметров. Переменные используют префикс `--` и организованы по категориям (цвета, типографика, размеры).

2. **Tailwind CSS:** Утилитарные классы для быстрой стилизации компонентов, настроенные на использование переменных из `theme.css`.

3. **Кастомные классы:** Дополнительные классы в `tailwind.css` для специфичных случаев, не покрываемых стандартным Tailwind.

4. **Темы:** Поддержка светлой и темной темы через атрибут `data-theme` и соответствующие переопределения переменных.

## Использование

При разработке новых фичей или компонентов всегда сначала проверяйте, нет ли уже подходящего решения в `shared`.

## План рефакторинга

На основе анализа существующей кодовой базы предлагается следующий план рефакторинга:

### 1. Стандартизация API для работы с Convex

1. **Создание унифицированных хуков**: Разработать набор хуков в `shared/hooks/convex`, которые обеспечат стандартизированный подход к:
   - Получению данных (`useConvexQuery`)
   - Выполнению мутаций (`useConvexMutation`)
   - Обработке ошибок и состояний загрузки

2. **Типизация API**: Расширить существующие типы в `shared/types` для более точного описания структур данных Convex.

### 2. Оптимизация стилей и UI-компонентов

1. **Аудит CSS-переменных**: Проверить и обновить CSS-переменные в `theme.css` для обеспечения консистентности.
2. **Рефакторинг компонентов**: Обновить компоненты для использования стандартных Tailwind-классов вместо инлайн-стилей.
3. **Создание компонентов высшего порядка**: Разработать HOC для часто используемых шаблонов UI (например, элементы с состоянием загрузки).

### 3. Улучшение структуры Effector-моделей

1. **Стандартизация API для моделей**: Создать единообразные паттерны для определения Effector-моделей.
2. **Оптимизация переиспользования**: Выделить общие паттерны в абстрактные модели и фабричные функции.
3. **Документирование интерфейсов**: Улучшить документацию API для каждой модели Effector.

### 4. Централизация управления состоянием

1. **Консолидация сторов**: Объединить разрозненные сторы в логические группы.
2. **Создание middleware**: Разработать middleware для логирования и отладки изменений состояния.
3. **Интеграция с DevTools**: Настроить интеграцию с Effector DevTools для упрощения отладки.

### 5. Тестирование и повышение надежности

1. **Создание тестовых утилит**: Разработать хелперы для тестирования Effector-моделей и UI-компонентов.
2. **Внедрение mock-сервисов**: Стандартизировать подход к мокированию Convex API для тестирования и разработки.
3. **Обработка краевых случаев**: Улучшить обработку ошибок и граничных условий в API-хуках.
